2. Assessment Requirements

This assignment is an exercise in test-based code refactoring (see Section 3.2). You may also employ the practice of pair programming [9,10] while refactoring your code (see Section 3.3). Using these practices, you will improve the quality of some pre-existing C++ code (see Section 3.5).

You are required to make use of the Git version-control software to manage your source code during development (see Section 3.4). You are also individually required to use a Jenkins continuous-integration server to monitor the automated test program in your Git repository (see Section 3.6).

2.1 Deliverables

You are required to:

  1. create a Git repository on Olympuss (see Section 3.6), and use this repository for code development;
  2. individually, create and configure Jenkins job(s) that monitor your repository;
  3. produce an updated version of the provided 'gridworld-route.cpp' C++ file;
  4. export a copy of your final Git log from your repository.

If you employ pair programming, then your source code will be shared with a partner. Nonetheless, both members of the pair should submit to the NOW dropbox folder:

3. Assignment Instructions

3.1 Context: NMEA Sentences

This assignment involves working with functionality that generates Global Positioning System (GPS) data as 'NMEA Sentences', which are a family of data formats structured according to a standard defined by the National Marine Electronics Association [1].

      The NMEA standard defines numerous sentence formats. While the official standard describing the individual sentence formats is not freely available, there are numerous sources of unofficial documentation available online (e.g. [2, 3]).

      3.2 Test-based Code Refactoring

      You are not required to develop the NMEA Generator functionality yourself — a complete definition of a C++ function that generates NMEA Sentences is provided for you. However, the quality of the code within the function definition is very poor. 

      Code refactoring is the act of modifying source code to improve its quality, without adding or changing any existing functionality. For this assignment, you are required to refactor the provided (poor-quality) definition of the function.

      A suite of Boost UTF automated tests is provided for this function, and all the tests currently pass. You should ensure that all of these tests still pass after you have refactored the function definition. Good practice is to run the tests after each refactoring step, to ensure that no bugs have been introduced into the code.

      3.3 Pair Programming [Optional]

      You are permitted to work in a pair and employ the software-engineering practice of pair programming [9,10]. However, if you prefer to work individually, then that is also permitted.

      You may arrange your own partner for this assignment. If there is an odd person leftover without a partner, then you may instead form a group of three. Note however that pair programming works best in pairs, so you should try to form pairs wherever possible. To facilitate finding a partner for this assignment, there is a 'Pairs for Referral 1' conversation thread in the 'Pair Programming' channel on Microsoft Teams.

      Note that if you work as a pair (or three), then you must employ the pair-programming practice. You are not permitted to work independently at different times. However, you may split from your partner mid-way through if you decide that you are not finding pair-programming productive, and work individually thereafter. You should not continue to work with a partner who is not contributing to the pair programming.

      3.5 Provided Codebase

      The repository contains a small C++ codebase supporting the construction of structured data representing geographical positions.

      The parts of the codebase that are the focus of this assignment are:

      • The 'GridWorld::Route::toNMEA()' function in the 'src/gridworld/gridworld-route.cpp' file.
      • A Boost UTF test suite, in the 'tests/gridworld/gridworld-route-tests.cpp' file.
      • A Qt project named 'NMEA_Generator-Tests.pro', providing the configuration for the automated test program.

      You are tasked with refactoring the (poor quality) definition of the 'toNMEA()' function.

      Clarifications:

      • You may make use of components from the C++ standard library, and may also use components from the Boost libraries.
      • You may define additional functions (or structs or classes), either within 'gridworld-route-tests.cpp' or elsewhere in the codebase.

      3.6 Using Jenkins to Monitor the Repository

      You are required to create and configure Jenkins job(s) that build and run the 'NMEA_Generator-Tests' program. After each commit, you should push to your remote Git repository on Olympuss so that Jenkins can record the result of the build/run.

      Note: to demonstrate your individual ability to use Jenkins, you are required to separately create and configure these jobs, even if you are working as a pair for the code refactoring.

      Additional instructions/clarifications:

      • The job(s) should be created on the 'tonga' server:
      https://tonga.ntu.ac.uk/
      • On tonga, create a folder named either as your student identifier or your name. Your job(s) should be created in that folder.
      • The job(s) should run on the 'Ganges' node.
      • You will need to instruct Jenkins to use the 'referral' branch of the Git repository.
      • You should not commit generated files to the Git repository for use by the job(s).
      • You should try to ensure that each run of the job(s) succeeds.

      For a high grade:

      • The jobs should be set up before you begin to refactor the C++ code.
      • There should be two jobs sharing a 'workspace':
        • one job to build the test program;
        • one job to run the test program;
      • The 'build' job should be automatically triggered whenever there is a push to the Git repository.
      • The 'run' job should be automatically triggered whenever the 'build' job completes successfully.

      6. Referencing, Plagiarism and Collusion

      For this assignment you may work in a pair (or, exceptionally, a group of three). If you do so, then your code will be jointly authored. Furthermore, you may discuss the use of software-engineering tools with students outside of your pair. However, you should not directly copy code written by another student outside of your pair for this assessment — that is collusion.

      Important:

      • You should not give another student (outside of your pair) access to your source code — if they then copy it, even without your permission, you will be considered to have colluded.
      • If you make a fork of the Git repository, then you should revoke access for any student teams that initially have access (if any).
      • If you created a new Git repository for this assignment (rather than making a fork), then ensure that repository is private!

      If you reuse or adapt several lines of code written by another person to solve a particular problem, then you should provide a reference to the source of that code — failure to do so is plagiarism. For this assignment, references can be provided as code comments in the 'gridworld-route-tests.cpp' file.

      8.3 Code Refactoring

      You are being assessed on your ability to refactor code in a test-based manner. Note that you must ensure that your submitted file compiles and allows the test program to build, and that the tests pass when run. This is because building and running the test program is an essential part of the test-based refactoring process.

      If tests fail after a refactoring step, then you should undo the change(s) until the tests are passing again.

      You are also being assessed on the quality of your refactored code. For this assignment, the main code-quality aspects are being assessed are as follows:

      • Choice of variable names (readability)
      • Appropriateness of code comments (understandability)
      • Naming constants (maintainability)
      • Program decomposition (modularity)

      8.3.1 Choice of variable names

      You should improve the variable names such that they indicate their role in the code. If you have introduced additional functions/structs/classes, then your choice of names for those components is also assessed.

      8.3.2 Code comments

      Unhelpful code comments should be removed or improved. Helpful comments should be added to explain any non-obvious parts of the code. However, you should not add comments that just restate the code without adding helpful explanation.

      8.3.3 Naming constants

      Program constants that are derived from the software requirements should be stored in named constant variables (or composite data structures for groups of related values), rather than having unnamed 'magic' literals in the program code (unnamed numbers, strings or characters that affect the program control-flow).

      8.3.4 Program Decomposition

      High-quality code should be organised into numerous small and simple functions, rather than a single large complex function.

      Grade Code Refactoring (Weighting 50%)
       2:1 Refactoring improvements that substantially restructure the entire function definition (e.g. consistently merging duplicated code into auxiliary functions).
       2:2 Refactoring improvements that substantially restructure isolated chunks of the code (e.g. using different language features / library components; introducing auxiliary functions; deleting unused code)
      High 3rd
      		 Refactoring improvements that partially restructure the code (e.g. introducing named constants; combining or splitting code across multiple lines; changing variable declaration locations).
      Low 3rd Refactoring improvements that indicate some understanding of the code logic (e.g. relevant improvements to variable names; helpful comments introduced).
      Marginal Fail
      		 Some tests fail when run OR refactoring improvements indicate misunderstandings of the code logic (e.g. changes to variable names, but new names are not indicative of their role; partially incorrect comments introduced)
      Mid Fail
      		 All tests fail when run OR refactoring improvements that are limited to those that do not require any understanding of the code logic (e.g. layout improvements; removal of old comments; correcting spelling of comments; namespace usage).
      Low Fail
      		 The test program does not build OR refactoring improvements are very limited.
      Zero
      		 No attempt has been made at refactoring the function.
